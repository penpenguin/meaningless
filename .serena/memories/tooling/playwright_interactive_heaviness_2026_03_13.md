2026-03-13: Investigated why the `playwright-interactive` skill feels heavy in the `meaningless` repo. Findings: (1) the skill itself is large (~689 lines / 31.6 KB) and prescribes a full QA workflow (inventory, exploratory pass, separate visual QA, screenshot artifacts), which is overkill for quick debugging. (2) The skill defaults to headed Chromium and encourages keeping desktop + mobile contexts alive; on this Three.js aquarium app that multiplies renderer cost. Measured against local Vite app: single desktop page `domcontentloaded` ~497 ms but `loadEventEnd` ~8.9 s; desktop+mobile together reached ~12.2 s and ~16.3 s loadEventEnd. (3) Chromium in this environment uses SwiftShader software rendering (`ANGLE ... SwiftShader`), so Playwright screenshot capture on the aquarium is extremely slow: full 1280x720 JPEG capture took ~46 s and even an 800x450 clipped capture took ~53 s; a default full-page screenshot timed out at 30 s. (4) Repo renderer is expensive by default: WebGLRenderer uses antialias=true, shadow maps enabled, and DPR up to 2; app also monitors FPS and auto-degrades quality when FPS < 30. Suggested mitigations: use a tiered workflow (quick/standard/signoff), default to one desktop context only, keep headless for routine checks, avoid screenshot-heavy QA except final signoff, and add a repo-specific low-quality QA mode or debug flag for Playwright runs.
